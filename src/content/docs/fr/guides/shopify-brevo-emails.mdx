## Architecture de la Solution

Pourquoi choisir l'API Transactionnelle de Brevo plutôt que les solutions intégrées ? 

La stack **Shopify Admin REST → Python → Brevo API v3** offre une flexibilité totale. Contrairement aux automatisations "no-code" qui facturent à l'exécution (comme Zapier), un script Python hébergé sur un VPS ou une Lambda AWS permet de traiter des milliers de commandes pour un coût quasi nul.

L'architecture repose sur un cycle de récupération (`Polling`) ou un déclencheur (`Webhook`) qui extrait les données de commande de Shopify pour les injecter dans un template Brevo hautement optimisé.

## Guide d'Implémentation Pas-à-Pas

### 1. Configuration des accès
Créez une application personnalisée dans votre administration Shopify. Vous devez accorder le scope `read_orders`. Récupérez le **Admin API Access Token**. Côté Brevo, allez dans vos paramètres pour générer une **Clé API v3**.

### 2. Développement du moteur de synchronisation
Le script suivant implémente une logique de production : gestion des erreurs, retry automatique en cas de rate-limiting et pagination réelle des ordres.

```python title="shopify_brevo_sync.py"
import os
import requests
import time
import re
from dotenv import load_dotenv
from requests.adapters import HTTPAdapter
from urllib3.util.retry import Retry

load_dotenv()

# Configuration des environnements
SHOPIFY_STORE_URL = os.getenv("SHOPIFY_STORE_URL")
SHOPIFY_ACCESS_TOKEN = os.getenv("SHOPIFY_ACCESS_TOKEN")
BREVO_API_KEY = os.getenv("BREVO_API_KEY")
BREVO_SENDER_EMAIL = os.getenv("BREVO_SENDER_EMAIL")

def get_session():
    """
    Configuration d'une session avec exponentially backoff.
    Crucial pour éviter les blocages 429 (Too Many Requests).
    """
    session = requests.Session()
    retry = Retry(
        total=5,
        backoff_factor=1,
        status_forcelist=[429, 500, 502, 503, 504],
        allowed_methods=["HEAD", "GET", "POST", "OPTIONS"]
    )
    adapter = HTTPAdapter(max_retries=retry)
    session.mount("https://", adapter)
    return session

def fetch_shopify_orders(session):
    """
    Récupération des commandes via l'API REST de Shopify.
    Gère la pagination via le header 'Link'.
    """
    orders = []
    # On cible les commandes récentes avec un limit de 50 par page
    url = f"{SHOPIFY_STORE_URL}/admin/api/2024-04/orders.json?status=any&limit=50"
    headers = {"X-Shopify-Access-Token": SHOPIFY_ACCESS_TOKEN}
    
    try:
        while url:
            response = session.get(url, headers=headers)
            response.raise_for_status()
            data = response.json()
            orders.extend(data.get("orders", []))
            
            # Analyse du header Link pour passer à la page suivante
            link_header = response.headers.get("Link")
            if link_header:
                match = re.search(r'<(.*?)>; rel="next"', link_header)
                url = match.group(1) if match else None
            else:
                url = None
    except requests.exceptions.HTTPError as e:
        print(f"Erreur Shopify API: {e.response.status_code}")
        raise e
    return orders

def send_brevo_email(session, customer_email, order_name):
    """
    Envoi de l'email via l'endpoint transactionnel de Brevo.
    """
    url = "https://api.brevo.com/v3/smtp/email"
    headers = {
        "api-key": BREVO_API_KEY,
        "Content-Type": "application/json"
    }
    
    # Payload structuré pour Brevo v3
    payload = {
        "sender": {"email": BREVO_SENDER_EMAIL, "name": "Ma Boutique Shopify"},
        "to": [{"email": customer_email}],
        "subject": f"Confirmation de commande {order_name}",
        "htmlContent": f"<html><body><h1>Merci !</h1><<pp>Votre commande {order_name} est validée.</p></body></html>"
    }
    
    try:
        response = session.post(url, json=payload, headers=headers)
        response.raise_for_status()
    except requests.exceptions.HTTPError as e:
        print(f"Erreur Brevo API: {e.response.status_code}")
        raise e

def main():
    session = get_session()
    try:
        orders = fetch_shopify_orders(session)
        for order in orders:
            email = order.get("email")
            name = order.get("name")
            if email:
                send_brevo_email(session, email, name)
    except Exception as e:
        print(f"Échec critique: {e}")

if __name__ == "__main__":
    main()
```

## Gestion des Erreurs et Sécurité

### Sécurisation des Clés API
L'erreur la plus commune est le "hardcoding" des accès. Utilisez systématiquement un fichier `.env` ajouté à votre `.gitignore`. En production, privilégiez les secrets GitHub Actions ou les variables d'environnement AWS/GCP.

### Le défi du Rate Limiting (Erreur 429)
Shopify et Brevo limitent le nombre de requêtes par seconde. Si vous traitez 10 000 commandes d'un coup, vous recevrez une erreur `429 Too Many Requests`. La solution implémentée ici est l'**Exponential Backoff** via `urllib3.util.retry`. Le script attend un temps croissant entre chaque tentative, garantissant que le flux ne sera pas coupé.

## Optimisation et Scalabilité

Pour passer d'une dizaine de commandes à un volume industriel (10k+ / jour), le polling (`GET /orders.json`) devient inefficace. 

**L'évolution logique : Les Webhooks.**
Au lieu de demander à Shopify "Y a-t-il des nouvelles commandes ?", Shopify envoie une notification `POST` à votre serveur dès qu'une commande est créée. Vous remplacez alors la fonction `main()` par un endpoint Flask ou FastAPI qui reçoit la donnée en temps réel et déclenche l'appel Brevo instantanément.
